SentientAI: From Prototype to Vercel Production

The "Production Pivot" Masterstroke

What makes this project unique is the response to critical deployment failure. While most students stop at "Model Accuracy," this project faced a real-world engineering bottleneck—Vercel's immutable resource limits—and solved it through total system re-engineering.

Critical Issues Identified

The Bottleneck Dependency Overload (800MB+)

The original Ultralytics/PyTorch stack exceeded serverless limits. Diagnostic analysis confirmed that the "one-size-fits-all" ML libraries were unsuitable for lightweight production hosting.

Architecture Conflict Scattered Asset Distribution

Model weights and training logs were duplicated across multiple directories (results/, static/results/, and root), causing confusion and unnecessary disk usage.

Performance Hurdle High Cold-Start Latency

The initial Flask setup was too "heavy," leading to slow initialization times that would be penalized in a serverless environment.

Solution: Engineering the Re-birth

Re-Engineering 01 Architectural Decoupling

By stripping the PyTorch legacy and manually implementing the inference logic in onnx_engine.py, the project transformed from a "prototype" into a "production engine."

Re-Engineering 02 Optimized Pipeline

Implemented custom Non-Maximum Suppression (NMS) and center-to-corner coordinate transformation to ensure zero loss in accuracy despite the 90% reduction in underlying weight size.

The Competitive Edge

Unlike standard academic projects, this project demonstrates Production-First Engineering. It proves the ability to diagnose root-cause deployment blockers, conceptualize a performance-focused pivot, and rebuild the core inference stack under strict resource constraints.
