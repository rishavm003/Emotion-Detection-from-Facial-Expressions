Project Evolution

Phase 1: Initial Research & Prototype

Training of a YOLO11s model on Kaggle GPU using the 9-class Emotion Expression dataset. The original architecture relied on the full Ultralytics framework and PyTorch, resulting in a >800MB environment overhead.

Phase 2: Local Web Integration

Development of the Flask backend and the "Cyber-Luxe" design system. Real-time inference was achieved using locally hosted `.pt` weights. System performed well on GPU, but CPU inference was sluggish due to heavy backend load.

Phase 3: Optimization & Production Redo

The system was completely re-architected for cloud deployment. The model was exported to ONNX format, and the backend was refactored to use onnxruntime. This reduced the bundle size by 95% and enabled serverless functionality.

Engineering Decisions: Why YOLO11s?

Traditional CNN or ViT (Vision Transformer) models often require a separate face-detection pre-stage. By choosing YOLO (You Only Look Once), we combine face detection and emotion classification into a single unified pass. This eliminates the latency of a separate pipeline, making real-time multi-face analysis possible even on a serverless CPU.

Optimization Insight: By switching from PyTorch to ONNX, we avoided the high memory overhead of the torch library, reducing the cold-start time from ~30s to < 2s.

Data Pipeline & Preprocessing

To achieve 87.3% mAP, we focused heavily on the internal data pipeline before the training phase.

  • Augmentation: Applied heavy horizontal flips, random brightness shifts, and rotations (±15°) to increase model robustness against varied lighting and camera angles.
  • Imbalance Handling: Handled dataset skew (e.g., Happy vs Contempt) using class-weighting in the loss function and oversampling minority classes.
  • Input Normalization: Images are letterboxed to 640x640 to preserve aspect ratio, followed by channel-wise mean/std normalization and fp32 cast.

Frontend & Scalability

The UI is built with Vanilla JavaScript to maximize performance and minimize bundle weight. It uses a custom state manager to track detection history without external library overhead.

Scalability Roadmap

  • Multi-User: Leverage Vercel's horizontal scaling (Lambda-per-request).
  • Batch Inference: API supports future batch uploads for bulk video analysis.
  • Load Balancing: Use Vercel Edge Network to route requests to the closest geographic region for lower latency.

Deployment Workflow

To deploy the current architecture from the feature/vercel-onnx-optimization branch:

1. GitHub Synchronization
git push origin feature/vercel-onnx-optimization
2. Vercel Project Setup

Link the repository to Vercel. Vercel automatically detects vercel.json and builds using the @vercel/python runtime.

3. Bundle Management

The best.onnx model is whitelisted in .gitignore to ensure it is bundled. Large legacy .pt files are stored in model/backups/ to keep the production bundle lean.

API Reference & Security

POST /detect

Primary endpoint for JSON-based frame inference. Supports base64 encoded JPG/PNG input.

Security & Hardening

  • Input Validation: Strict MIME-type checking and resolution capping (640px) to prevent buffer overflows or DoS.
  • Rate Limiting: (Planned) IP-based request throtelling to prevent API abuse.
  • No Persistence: Frames are processed in-memory and discarded instantly to ensure user privacy.
